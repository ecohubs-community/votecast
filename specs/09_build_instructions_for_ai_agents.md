# Community Governance Platform — Build Instructions for AI Coding Agents

# 1. Purpose

This document defines the **recommended implementation order and development workflow** for building the Community Governance Platform.

AI coding agents should follow these instructions to avoid:

- building components in the wrong order
- introducing architectural inconsistencies
- creating circular dependencies

The system must be implemented **incrementally**, ensuring that each layer works before the next layer is added.

---

# 2. High-Level Implementation Order

The platform must be built in the following order:

```
1. Project initialization
2. Database schema
3. Core backend services
4. Event system
5. Authentication
6. API endpoints
7. Basic frontend
8. Voting interface
9. Plugins and hooks
10. Final UI improvements
```

Each phase must compile and run successfully before moving to the next.

# 4. Step 2 — Database Setup

Database dependencies are already installed (`drizzle-orm`, `better-sqlite3`).

Database connection module exists at:

```
/src/lib/server/db/index.ts
```

**Important:** The current schema (`src/lib/server/db/schema.ts`) contains a demo `task` table that must be replaced with the actual governance schema.

The `auth.schema.ts` file is auto-generated by better-auth and must be preserved.

Tables required for MVP (see 03_database_schema.md for full definitions):

```
users (managed by better-auth + custom fields)
communities
community_members
invites
proposals
proposal_choices
votes
webhooks
execution_handlers
```

Note: better-auth manages its own `user`, `session`, and `account` tables. The governance `users` table may be the same as better-auth's `user` table with additional fields, or a separate table linked via foreign key. Prefer extending better-auth's user table.

Run initial migration after schema is defined.

# 6. Step 4 — Event System

Implement the event bus.

Location:

```
/src/lib/server/events
```

Core functions:

```
emit(event, payload)
subscribe(event, handler)
```

Then integrate event emission into services.

Examples:

```
proposal-service → emit("proposal.created")
vote-service → emit("vote.cast")
```

# 7. Step 5 — Authentication System

Authentication uses **better-auth** framework (already configured at `src/lib/server/auth.ts`).

Session middleware is already implemented in `src/hooks.server.ts`.

Remaining tasks:

```
1. Add wallet-based login as a better-auth provider/plugin
2. Configure nonce generation for wallet signature verification
3. Ensure session middleware attaches user + session to event.locals
4. Add email/password as fallback authentication (already enabled)
```

The wallet authentication flow (SIWE-style) should be implemented through better-auth's plugin system rather than as a custom standalone module.
